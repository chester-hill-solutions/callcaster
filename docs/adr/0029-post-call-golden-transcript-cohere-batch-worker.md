# Post-call golden transcript via Cohere Transcribe batch (worker job)

After a call ends and the recording is persisted to Railway Buckets (`call.audioUrl`, written by the `/api/recording` Bun route per ADR-0027), a **Bun worker job** (`type='cohere_batch_transcribe'`, per ADR-0007) re-transcribes the recording with **Cohere Transcribe** (`cohere-transcribe-03-2026`, 2B Conformer, 14 languages, 5.42% WER on LibriSpeech, 3× faster than peers on clean audio). The result is stored as a second `call_transcript` row with `provider='cohere_batch'` and `call.transcriptId` is repointed at it, replacing the live Deepgram transcript in post-call views. Cohere is the **golden post-call transcript** because: (a) it's more accurate than Deepgram Nova-3 on clean post-call audio (Deepgram is tuned for telephony streaming, Cohere wins on the de-noised recording), (b) it's cheaper (Cohere API free tier for low-setup, Model Vault per-hour-instance for production without rate limits), (c) it's multilingual across 14 languages vs Deepgram's per-model language selection, and (d) batch transcription has no latency constraint so the file-upload multipart API (`POST /v2/audio/transcriptions`, 25MB limit, `flac/mp3/mpeg/mpga/ogg/wav`) is fine. The Deepgram live transcript remains the source for the in-call UI; Cohere replaces it in the call-detail view when available. The worker scheduler enqueues the job every 15 minutes for calls where `audioUrl IS NOT NULL AND transcriptId IS NULL AND createdAt > now() - interval '24 hours'`, with missed-cron catch-up on boot (ADR-0007). The job handler validates params with Zod (`{ callId: number }`), downloads from Railway Buckets, chunks with ffmpeg if >25MB (ffmpeg is in the worker Docker image), calls Cohere with `language` from `workspace.coachingConfig` (default `en`), inserts the `call_transcript` row, updates `call.transcriptId`, and debits 1 credit via the idempotent ledger (key `transcription_batch:${callId}`). Cohere Transcribe's limitations — no native streaming, no diarization, no timestamps — are acceptable for the post-call path because the recording is a complete file and the live Deepgram transcript already has speaker labels and timestamps for the in-call UI. This split (Deepgram live + Cohere batch) uses each provider where it wins and is why a single-provider approach was rejected.

## Considered Options

- **Deepgram for both live and post-call** — simpler stack, one vendor, one SDK. Rejected because Cohere is materially more accurate on clean audio (the post-call recording is de-noised vs the live 8kHz stream) and meaningfully cheaper at batch volumes. The accuracy delta matters for the golden transcript that gets stored permanently and reviewed post-call.
- **Cohere for both (self-hosted streaming)** — would unify the vendor but Cohere has no streaming endpoint. Self-hosting on vLLM with chunked inference adds GPU pod infrastructure, has no diarization/timestamps out of the box, and is weaker on phone audio. The complexity isn't worth vendor unification.
- **Twilio's built-in `transcribe: true`** — v1 used this (`call.action.server.ts:86`) with a `transcribeCallback` to a `/api/transcribe` route that never existed (Twilio callbacks 404'd on every outbound call). Lower quality than either Deepgram or Cohere, locks transcription to Twilio's model choices, and the callback was broken. Not carried into v2.
- **No post-call batch (live Deepgram only)** — the live transcript has telephony-streaming artifacts (no punctuation recovery, no re-decoding with wider context). The golden transcript is the one that gets shown in the call-detail view, exported in analytics, and used for the `coaching_session.summary`. Using the live transcript for permanent storage means shipping a lower-quality artifact.

## References

- Cohere Transcribe: `cohere-transcribe-03-2026`, 2B Conformer encoder-decoder, Apache 2.0, `POST https://api.cohere.com/v2/audio/transcriptions` (multipart: `model`, `language` ISO-639-1, `temperature`, `file`), 25MB max, returns `{"text": "..."}`. Available via API (free tier, rate-limited) or Model Vault (per-hour-instance, no rate limits).
- Cohere limitations (relevant to why it's batch-only): no native streaming/WebSocket endpoint, no speaker diarization, no timestamps. Hugging Face `CohereLabs/cohere-transcribe-03-2026` supports `transformers`, `vLLM`, `mlx-audio`, `transformers.js` (browser) — but no first-party streaming server.
- Telephony accuracy: April 2026 benchmarks (dev.to/jim_l_efc70c3a738e9f4ba) — "Phone audio, telephony, voicemail: Whisper-large-v3 still has the edge from telephony augmentation" over Cohere; Deepgram Nova-3 is purpose-built for 8kHz telephony streaming and accepts mulaw natively.
- ADR-0007 (worker + job table — `cohere_batch_transcribe` job type, scheduler entry, missed-cron catch-up, Zod param validation, `@chs/job-worker` claim pattern)
- ADR-0027 (live transcription — the `/api/recording` fix that persists `call.audioUrl` is the hard prerequisite; without the recording in Railway Buckets, there's nothing to batch transcribe)
- v1 dead code (not copied to v2): `transcribe: true` + `transcribeCallback: ${baseUrl}/api/transcribe` on `call.action.server.ts:86-87,107-108` pointed at a nonexistent route; `ivr-recording/index.ts:278` hardcoded `transcription: null` on IVR recordings; inbound voicemail `<Record>` used `transcribe: true` with no `transcribeCallback` so Twilio's transcription was never retrieved.
- `@chs/job-worker` (shared package — claim loop, HTTP wake, scheduler, idempotency key via SHA-256 hash)
