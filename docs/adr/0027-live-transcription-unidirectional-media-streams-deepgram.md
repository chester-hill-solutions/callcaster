# Live transcription via unidirectional Media Streams sidecar + Deepgram Nova-3

Live in-call transcription is built as a **unidirectional Twilio Media Streams sidecar** — Twilio forks both audio legs (`<Start><Stream track="both_tracks">`) to a WebSocket endpoint on the media-stream Bun service (ADR-0030), which forwards the 8kHz mulaw frames directly to **Deepgram Nova-3 streaming** and publishes diarized transcript segments to the agent's browser via pg-realtime SSE (ADR-0005). The AI never speaks on the call — bidirectional Media Streams (`<Connect><Stream bidirectional>`) are explicitly rejected because they add barge-in, `mark`-event bookkeeping, a TTS pipeline, and ~2-3× the wiring complexity for a coaching sidecar that only needs to listen. Deepgram Nova-3 is chosen over Cohere Transcribe for the live path because: (a) Cohere Transcribe has no native streaming/WebSocket endpoint (it's a batch file-upload model, 25MB multipart — see ADR-0029 for the post-call path where Cohere wins), (b) Deepgram accepts 8kHz mulaw natively with zero resampling (Twilio's wire format), (c) Deepgram's `diarize_model=latest` separates agent from contact speaker in real time (Cohere has no diarization), (d) `filler_words=true` tags "uh"/"um"/"mhmm" out of the box (the live-coaching primitive, ADR-0028), and (e) `utterance_end_ms=1000` + `vad_events=true` give pause detection for free. The sidecar is gated per-workspace by `workspace.featureFlags.liveTranscription` (Zod-typed jsonb column, ADR-0003) — workspaces without the flag get the standard `<Dial>`/`<Conference>` TwiML with no `<Stream>` element, so the feature is opt-in and zero-cost when off. Transcripts persist as `transcript_segment` rows (one per Deepgram `speech_final` utterance, with `speaker`, `startMs`, `endMs`, `fillerCount`) and a `call_transcript` row with `provider='deepgram_live'`; both reference `call.id` (domain PK, ADR-0015), not the Twilio SID. The recording webhook (`/api/recording` Bun route, ADR-0009) downloads the Twilio-hosted recording to Railway Buckets (ADR-0002) and writes `call.audioUrl` — this is the hard prerequisite for the post-call Cohere golden transcript (ADR-0029). The v1 dead `transcribe: true` + `transcribeCallback` TwiML attributes (which pointed at a nonexistent `/api/transcribe` route) and the stub `/api/recording` route do not exist in v2's clean rebuild.

## Considered Options

- **Bidirectional Media Streams (`<Connect><Stream bidirectional>`)** — required only if the AI speaks on the call (whisper coaching, AI voice agent). Adds barge-in detection via `mark` events, a TTS pipeline (Deepgram Aura / ElevenLabs), `clear`-on-interrupt logic, and ~2-3× the wiring. The caller would hear the AI. Explicitly deferred — unidirectional covers ~95% of coaching value (transcript + on-screen cues + post-call report) at a fraction of the complexity. Re-evaluate if supervisor whisper/barge (ADR-0012 M3 milestone) needs an AI voice leg.
- **Twilio Conversation Relay** — Twilio's first-party STT+TTS. Easiest wiring (native Twilio, no external WS), but locks transcription to Twilio's model choices, is designed for AI voice agents (not human-agent coaching sidecars), and doesn't expose filler-word tags or per-utterance metrics. Also couples a billable feature to a single vendor's model roadmap.
- **Cohere Transcribe self-hosted streaming (vLLM chunked)** — would unify the live + batch provider, but Cohere ships no streaming endpoint (only batch multipart upload). Self-hosting `cohere-transcribe-03-2026` on vLLM with chunked inference is possible but adds GPU pod infrastructure, has no diarization or timestamps out of the box, and is weaker on phone-bandwidth audio than Whisper-large-v3 (per April 2026 benchmarks). Telephony audio is exactly Deepgram's training domain.
- **Browser-side transcription (Web Speech API)** — the Twilio Voice SDK does not expose the remote caller's audio as a tap-able `MediaStream`; only the local mic is accessible. Client-side transcription of the contact is not possible without Media Streams.
- **Cohere Transcribe via batch only (no live)** — defer all transcription to post-call. Loses the live coaching use case (ADR-0028), which is the headline feature.

## References

- Twilio `<Stream>` TwiML: `<Start><Stream track="both_tracks" url="wss://...">` is non-blocking (Twilio forks audio then continues to the next verb), so it cleanly precedes `<Dial>`/`<Conference>` in every call-mode route
- Deepgram Nova-3 streaming: `model=nova-3`, `language=en-US`, `smart_format=true`, `diarize_model=latest`, `filler_words=true`, `punctuate=true`, `utterances=true`, `utterance_end_ms=1000`, `vad_events=true`, `encoding=mulaw`, `sample_rate=8000`, `channels=1`
- v1 dead code (not copied to v2): `app/routes/api+/call.action.server.ts:86-87,107-108` (`transcribe: true` + `transcribeCallback: ${baseUrl}/api/transcribe` — the `/api/transcribe` route never existed, Twilio callbacks 404'd), `app/routes/api+/recording.action.server.ts:9-35` (stub that validated Twilio signature + logged but never downloaded the recording or wrote `call.recording_url`/`sid`/`duration`), `app/AudioStreamer.tsx` (dead prototype opening `ws://localhost:3000`, never imported), `supabase/functions/call-server/index.ts` (hello-world placeholder)
- ADR-0005 (pg-realtime SSE — transcript segments are unidirectional, event-shaped, resume-valuable; `Last-Event-ID` cursor resume means no lost transcript on reconnect, strictly better than the v1 Supabase Realtime broadcast which dropped on disconnect)
- ADR-0012 (conference-per-call — if v2 unifies all outbound on conferences, the stream fork always comes from the conference, simplifying wiring)
- ADR-0015 (`call.id` domain PK, `call.twilioSid` correlation column — TwiML custom `<Parameter>` carries `callId`, not `callSid`)
- ADR-0029 (post-call Cohere batch golden transcript — depends on this ADR's `/api/recording` fix to persist `call.audioUrl`)
- ADR-0030 (media-stream Bun service — the WebSocket endpoint that Twilio connects to)
