# Media-stream Bun service as third Railway process

The Twilio Media Streams WebSocket bridge + Deepgram streaming client + coaching engine run in a **third Railway service** alongside the web (Bun.serve + React Router) and worker (Bun job-processor) services ŌĆö all three are the same codebase with different entry points, consistent with ADR-0001 (Bun single runtime, multiple processes) and ADR-0007 (worker is same binary, different CMD). The media-stream service uses `Bun.serve({ port: 8081, websocket: { ... } })` ŌĆö Bun has native WebSocket support, so no `ws` library is needed (unlike v1 which would have required `ws@^8.16.0` on Express). The service exposes `/media-stream` (WS upgrade, where Twilio connects and streams 8kHz mulaw frames as base64 JSON) and `/healthz` (Railway healthcheck). On a Twilio `start` message, it opens a Deepgram Nova-3 streaming WebSocket with the custom parameters from the TwiML (`callId`, `workspaceId`, `userId`, `contactId`, `campaignId`); on each `media` message it decodes the base64 mulaw and forwards it to Deepgram; Deepgram returns diarized, filler-tagged, utterance-ended transcripts which the coaching engine (ADR-0028) processes; results are written to Postgres via the **admin Drizzle client** (cross-workspace, like the worker ŌĆö the scoped `createTenantDb` is not used here because the service is not request-scoped to a workspace) and published to pg-realtime via `workspace_events` insert + `pg_notify` in the same transaction (ADR-0005). On `stop`, the service finalizes the `coaching_session` row, closes the Deepgram connection, and debits transcription credits via the idempotent ledger (ADR-0006). Isolating this in a third service (rather than attaching the WS upgrade to the main Bun web server) keeps audio-processing CPU and Deepgram-buffer memory off the request-handling path ŌĆö a concurrent-call spike on the media-stream service cannot add latency to web requests or Twilio webhook processing. The service autoscales by concurrent WebSocket connections on Railway. The public WSS URL (`MEDIA_STREAM_URL`) is separate from `BASE_URL` and is what Twilio's `<Stream url="wss://...">` connects to.

## Considered Options

- **Attach WS upgrade to the main Bun web server** ŌĆö `Bun.serve` can handle WS upgrades on the same port as HTTP. One process, one deploy. Rejected for production because a Deepgram-buffer spike or CPU-heavy coaching-loop iteration on a concurrent call could add latency to web requests and Twilio webhook handling. The web tier must stay responsive for call-status callbacks. The split is cheap (same codebase, different entry point, same Docker image) and the isolation is valuable.
- **Sibling service as separate repo** ŌĆö rejected; the service imports Drizzle schema, `shared/billing-rates.ts`, and `@chs/*` packages from the callcaster monorepo. A separate repo would require publishing these as packages or vendoring them. Same-codebase-different-entrypoint matches the worker pattern (ADR-0007).
- **Cloudflare Workers / Durable Objects** ŌĆö rejected. Introduces a different runtime and vendor into a Bun-on-Railway stack with zero Cloudflare footprint. Durable Objects would be the right primitive for per-call state coordination on Cloudflare, but the entire v2 plan (ADR-0001, ADR-0002) is Bun-single-runtime on Railway. The media-stream service holds per-call state in a `Map` in memory (the same way the worker holds job state); a call's WS connection is a single long-lived process by nature.
- **Supabase Edge Function (v1 pattern)** ŌĆö rejected. Deno Edge Functions are request-scoped with lifetime limits and cannot hold a bidirectional WebSocket for the duration of a call. v2 sheds Supabase entirely (ADR-0002). The v1 `supabase/functions/call-server/index.ts` was a dead "Hello world" placeholder, never a real WS server.

## References

- Bun WebSocket API: `Bun.serve({ port, websocket: { open(ws), message(ws, msg), close(ws) }, fetch(req, server) { server.upgrade(req) } })` ŌĆö native, no `ws` dep
- Twilio Media Streams WebSocket protocol: `connected` ŌåÆ `start` (with `streamSid`, `callSid`, custom `<Parameter>` values) ŌåÆ `media` (20ms mulaw frames, base64, ~50/sec) ŌåÆ `stop`. Server sends only keepalives in unidirectional mode (no `media`/`mark`/`clear` back to Twilio).
- Deepgram streaming: `wss://api.deepgram.com/v1/listen?model=nova-3&...` ŌĆö KeepAlive JSON every 5s to prevent timeout, binary audio frames in, JSON `Results` messages out (`channel.alternatives[0].transcript`, `channel.alternatives[0].words[]` with `start`/`end`/`confidence`, `speech_final`, `is_final`)
- ADR-0001 (Bun single runtime ŌĆö this service is a third entry point on the same binary)
- ADR-0004 (scoped vs admin Drizzle client ŌĆö media-stream uses admin client, not `createTenantDb`, because it's not request-scoped)
- ADR-0005 (pg-realtime ŌĆö `workspace_events` insert + `pg_notify('workspace_event', eventId)` in the same Drizzle transaction as the `transcript_segment`/`coaching_event` insert)
- ADR-0006 (billing ŌĆö credit debit is a Drizzle tx: idempotent ledger insert + `SELECT FOR UPDATE` credits update)
- ADR-0007 (worker ŌĆö same "same codebase, different entry point" pattern; `@chs/job-worker` claim pattern reused for any internal job the media-stream service enqueues)
- ADR-0009 (Twilio webhooks into Bun ŌĆö the `<Start><Stream>` TwiML is generated by Bun routes; the WS bridge receives the stream)
- v1 dead code (not copied to v2): `scripts/dev/websocket-server.js` (dev-only WSS echo server, not wired into the app), `app/AudioStreamer.tsx` (dead client-side prototype opening `ws://localhost:3000`), `supabase/functions/call-server/index.ts` (hello-world placeholder)
